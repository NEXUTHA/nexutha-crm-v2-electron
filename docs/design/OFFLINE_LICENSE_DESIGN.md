# 設計書：脱Supabase オフラインライセンス（台帳 L-01 / L-02 / L-03）

**作成**: 2026-07-20 / **対象リリース**: v2.5.2 再カット / **状態**: 承認待ち

---

## 1. 何を解決するか

現在、ライセンス検証は Supabase へのネットワーク照会のみで（`index.html:6103`）、そのサーバーが消滅したため**全ユーザーが起動できません**。ネットワークに依存しない仕組みに作り替えます。

**方針**：アプリには**公開鍵だけ**を埋め込み、キーの正しさを**その場で数学的に検証**します。サーバーは不要になります。

---

## 2. 全体像

```
【山下さんの手元（オフライン）】
  秘密鍵（Keychainに保管・門外不出）
        │
        │  node tools/make-license.js --name "〇〇株式会社" --type normal
        ↓
  ライセンスキー（テキスト）───メール等で顧客へ───┐
                                                    │
【顧客のアプリ】                                    ↓
  公開鍵（アプリに同梱）── で検証 ──── キーを入力
        ↓
  数学的に正しければ通過（ネットワーク一切不要）
```

**署名方式**: Ed25519（楕円曲線デジタル署名）。秘密鍵を持つ人だけが有効なキーを作れ、公開鍵を持つ人は検証だけできます。**公開鍵からは秘密鍵を逆算できません**ので、アプリを配布しても偽造されません。

---

## 3. キーの形式

### ペイロード（署名される中身・16バイト）

| 項目 | サイズ | 内容 |
|---|---|---|
| バージョン | 1 | 形式の版（`1`） |
| 種別 | 1 | `1`=master（山下さん用） / `2`=normal（販売） / `3`=trial |
| 発行日 | 2 | 2026-01-01からの日数 |
| 通し番号 | 4 | 発行ごとに +1 |
| **有効期限** | 2 | 同上の日数。**`0` = 無期限**（v1は常に0。将来のサブスク用に枠だけ確保） |
| 予備 | 6 | 将来拡張用（`0`埋め） |

### 全体

```
[ペイロード 16バイト] + [Ed25519署名 64バイト] = 80バイト
        ↓ Base32（Crockford・誤読しにくい文字集合）
128文字 → 5文字ずつハイフン区切りで表示

例（形式のみ。実際の値ではありません）:
NX2-A4K7M-P9RT2-...（全26グループ）
```

### ⚠️ 正直な注意：キーは長くなります

**Ed25519の署名は64バイト固定で、これは短くできません**（切り詰めると安全性の証明が壊れます）。結果、キーは**約128文字**になります。**手で打つ長さではありません。**

前提として、**キーはメールやファイルで渡し、顧客はコピー＆ペーストする**運用になります。入力欄は、貼り付けたテキストから空白・改行・ハイフンを自動で取り除き、小文字も受け付けるようにします（打ち間違い耐性）。

### 発行先名について — ご判断ください

指示書では「ペイロードに発行先名を含める」とありましたが、**2案あります。**

| | 案A：名前をキーに入れる | 案B：通し番号のみ（**推奨**） |
|---|---|---|
| キーの長さ | 約155文字 | **約128文字** |
| 誰のキーか | キー自体から分かる | `tools/licenses.csv`（山下さんの手元の台帳）で通し番号から引く |
| 顧客名の扱い | **キーに顧客名が埋まる**。キーが第三者に渡ると顧客名も漏れる | キーに個人情報を含まない |
| アプリの表示 | 「〇〇株式会社 様のライセンス」 | 「ライセンス #0042（通常）」 |

**案Bを推奨します。** 短く、個人情報を含まず、発行台帳は手元にあるので誰のキーかは確実に分かります。ただし「アプリに顧客名を表示したい」というご希望があれば案Aにします。

## 4. どこで検証するか

**Electronの司令塔（`main.js`）で検証します。** 画面側からは `preload.js` 経由で依頼するだけです。

理由：
- Node.js の標準機能（`crypto.verify`）が Ed25519 に確実に対応しており、**追加ライブラリが不要**
- 画面側の WebCrypto は Ed25519 対応がブラウザ版に依存し、確実性に欠ける
- Python側で行うと `cryptography` ライブラリが必要になり、配布物が重くなる

```
画面(index.html)  ──window.electronAPI.verifyLicense(key)──▶  preload.js
                                                                  │ IPC
                                                                  ▼
                                                    main.js（公開鍵で署名検証）
                                                                  │
                  ◀────────── true / false ──────────────────────┘
```

公開鍵は `main.js` に定数として埋め込みます（公開鍵なので秘匿不要）。

## 5. 既存ユーザーの保護（重要）

**すでに認証済みの端末は、そのまま通します。**

現在の `submitLicense`（`index.html:6146`）は、**検証に成功したときだけ**キーを保存します（`:6155` → `saveLicenseKey`）。つまり **`settings.license_key` にキーが入っている ＝ 過去にSupabaseの検証を通過した証拠**です。

そこで：

1. 起動時、保存済みキーがあれば**まず新方式で検証**
2. 新方式で通らない場合、**旧形式のキーが保存されている**なら「移行済み」フラグ（`settings.license_grandfathered = 1`）を立てて**通過させる**
3. 以後はフラグを見て通す

これにより、**アップデートしただけの既存ユーザーは何も入力せずに使い続けられます。**

> ⚠️ **ただし、購入者が再インストールした場合は保存済みキーが無いため通りません。** 購入者には**新方式のキーを新規発行して送る必要があります**（再カット出荷後の連絡に添付）。

## 6. 秘密鍵の扱い

**これが最重要の運用ルールです。秘密鍵が漏れると、誰でも無限にライセンスを作れます。**

| 項目 | 方法 |
|---|---|
| 生成 | `node tools/keygen.js` をオフラインで1回だけ実行 |
| 保管① | macOSキーチェーン：`security add-generic-password -s nexutha-license-signing -a nexutha -w "<秘密鍵>"` |
| 保管②（バックアップ） | **紙に印刷 or USBメモリ**に書き出し、**VAULTとは別の場所**に物理保管 |
| リポジトリ | **絶対に入れない**。`.gitignore` に `tools/*.key` `tools/licenses.csv` を追加 |
| アプリ | **絶対に入れない**（公開鍵のみ同梱） |
| チャット | **絶対に出力しない**（生成も発行も山下さんが実行し、値は私に見せない） |

発行CLIはキーチェーンから秘密鍵を読み、**ファイルには一切書きません**。

### 発行の手順（山下さんが実行）

```
node tools/keygen.js                          # 初回のみ。鍵ペアを作る
node tools/make-license.js --type master      # 山下さん用（マスターキー）
node tools/make-license.js --type normal --name "〇〇株式会社"   # 顧客用
```

出力はキー文字列1行と、`tools/licenses.csv` への追記（通し番号・種別・発行日・宛先）。

**マスターキーはコード上の特別扱いをしません。** 種別が `master` の、ごく普通の署名済みキーです。分岐を作らないことで「特別なキーだけ通る抜け道」を作らずに済みます。

## 8. v1の割り切り（★ご判断ください★）

**この方式は、アプリを改造されれば回避できます。**

検証はお客様のMac上で動くため、技術的な知識がある人が DevTools（`Cmd+Alt+I`。`main.js:602-604` で配布版でも開けます）や、アプリファイルの書き換えによって「検証を通ったことにする」ことは可能です。

**v1ではこれを許容し、難読化などの対策は行いません。** 理由：

- 難読化は**時間稼ぎにしかならず**、本気の解析は防げません
- 対策のコストが、想定される被害（個人事業主向けの業務ソフトの不正利用）に見合いません
- **今の最優先は「正規のお客様が使えない」状態の解消**であり、不正対策ではありません

**現状より悪化はしません**（現在のSupabase方式も、応答を偽装すれば同様に回避できました）。

**この割り切りでよいか、承認時にご判断ください。** 対策を入れる場合は v2.5.3 以降で、DevToolsの無効化（台帳 L-06）や検証結果の署名保存などを検討します。

## 9. できないこと（正直に）

| 項目 | 状況 |
|---|---|
| **失効（解約対応）** | **できません。** オフライン検証は「発行済みキーを後から無効化する」手段を持ちません。返金・解約時にキーを止められません |
| 台数制限 | できません（同じキーで何台でも使えます） |
| 有効期限 | **形式上の枠は用意しますが、v1では常に無期限**（`0`）。将来サブスク化するときに使えます |
| 利用状況の把握 | できません（サーバーに繋がないため） |

**将来サブスクを本格化するなら、「オフラインで動くが、時々オンライン確認して猶予期間を延長する」ハイブリッド方式が必要になります。** これは v2.6 以降の課題として台帳 L-05 に登録済みです。
