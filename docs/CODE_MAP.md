# NEXUTHA CRM コード全体地図

**読む人**: オーナー（山下さん） / **作成**: 2026-07-20 / **対象**: `ba64b3b`

このアプリがどう作られているか、どこに何があるかを、専門用語をできるだけ使わずにまとめたものです。

---

## 0. まず全体像を一言で

このアプリは、**3つの部品が重なってできています。**

```
┌─────────────────────────────────────────┐
│ ① 外側の入れ物（Electron / main.js）     │  ← ウィンドウを開く・裏方を起動する
│  ┌───────────────────────────────────┐  │
│  │ ② 画面（index.html ほか）          │  │  ← 見た目と操作のほぼ全部
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
              ↕ やりとり
┌─────────────────────────────────────────┐
│ ③ 裏方（Python / backend/app.py）        │  ← データの保管庫（SQLite）
└─────────────────────────────────────────┘
```

**②の画面が全体の9割を占めています。** `index.html` 1ファイルに6,842行あり、そのうち4,804行がプログラムです。ここが良くも悪くもこのアプリの中心です。

## 1. ファイルごとの役割

### ① 外側の入れ物

| ファイル | 役割 |

## 2. データの流れ — ダブルクリックから顧客一覧まで

```
① アイコンをダブルクリック
     ↓
② main.js 起動：二重起動チェック(31-35) → ポート決定(159)
     ↓
③ 準備開始(561)：インストール場所の修復(564) → キャッシュ整理(566)
                 → メニュー作成(568-583) → ポート掃除(595)
     ↓
④ ウィンドウを開き、ぐるぐる画面を表示(596→345-422, 367)
     ↓
⑤ 裏方(Python)を起動(597→205-246)
     ↓
⑥ 裏方が立ち上がる：DBの場所を決め(app.py:22-46) → 表を作り(59-230)
                   → 9876番で待ち受け開始(1294)
     ↓
⑦ 「準備できた?」を1秒ごとに最大60回確認(main.js:271-292 → app.py:344-346)
     ↓
⑧ 画面を読み込む(main.js:316 → app.py:321-323 が index.html を返す)
     ↓
⑨ 画面が読み込まれる
     ★ CDNから4つの部品をダウンロード(index.html:30-33)  ← 要インターネット
       style.css(35) → state.js → renderer.js → pdf.js(66-68) ← 順序固定
       → 巨大プログラム(1931-6734)
     ↓
⑩ 読み終わり(6336)：設定を復元(6347-6353) → 1.2秒後にぐるぐるを消す(6360-6362)
     ↓
⑪ データ取得(6365-6366)：GET /api/customers → app.py:351-362 → SQLite
     ↓
⑫ ★ライセンス確認(6368→6140→6103)  ← 要インターネット（現在ここで全員止まる）
     NGなら画面を出して以降を全部中止(6369-6373)
     ↓
⑬ 初回セットアップ確認(6376→6275)
     ↓
⑭ ダッシュボード表示(6379→2274→renderer.js:7)
```

### ★ 顧客一覧が出るまでに必ず通る「関所」は3つ

1. **裏方の起動待ち**（最大60秒）
2. **CDNからの部品ダウンロード**（インターネット必須）
3. **ライセンス認証**（インターネット必須）

**どれか1つでも折れると画面は出ません。** 現在は3が完全に折れており（サーバー消滅）、2もオフラインでは折れます。

## 5. 触ると壊れやすい場所 トップ5

### 第1位 🔴 CDNからの4ライブラリ（`index.html:30-33`）
ローカルコピーが**存在しません**。`index.html:2044` の `new Dexie(...)` はプログラムの最上部で無条件に動くため、ダウンロードに失敗すると**その下の4,700行が1行も定義されず**、アプリが完全に沈黙します。皮肉なことに **Dexie は実際にはほとんど使われていません**（`USE_BACKEND=true` のため）。**使っていない部品のせいで全損する構造**です。

### 第2位 🔴 `index.html` の巨大プログラムと `js/*.js` の相互依存
`js/renderer.js` と `js/pdf.js` は、`index.html` の中で定義される道具（`handleError`・`showToast`・`API_BASE`・`NAPI`・`calcDocTotal` など）に**全面的に依存**しています。逆に `index.html` も `js` 側の関数を呼びます。**読み込み順（`index.html:66-68`）が変わると壊れます。**

特に危険な1行：`js/pdf.js:375` と `:632` は `allDocuments` という**裸の名前**を使っており、これは `index.html:2179` の「宣言なしの代入」が偶然作った変数に依存しています。**掃除の対象に見える書き方なので、消すとPDF再発行とプレビューが壊れます。**

### 第3位 🔴 `release.yml` の処理順序
`公証 → staple → ハッシュ計算 → yml生成` の順序が絶対です（`release.yml:105-153`）。`stapler staple` はファイルの中身を書き換えるため、**ハッシュを先に計算すると自動更新が永久に適用されません**。過去に実際この障害が起きています（`version.json` の 2.4.6 / 2.4.7 の履歴）。ファイル内に「★この後dmgのバイトが変わる★」「順序を厳守」と警告コメントがあります。

### 第4位 🟠 `main.js` の裏方管理
- ポート9876を使う**他人のプロセスを名前確認せず強制終了**します（`main.js:169-187`）
- 裏方が **`0.0.0.0` で待ち受け＋認証なし**（`app.py:1294`, `:51-57`）＝**同じWi-Fiの誰でも全顧客データを読み書きできます**
- `ai-chat.html` が配布物に入っておらず、配布版でAI機能が動きません

### 第5位 🟠 データベースまわり
- 顧客を消しても取引・ファイル・名刺が**孤児として残る**（外部キー制約が無効のまま）
- 書類更新で **「0」や「空」に修正できない**（`app.py:1018-1041`）
- 書類更新で **銀行情報が消える**（`app.py:1029` だけ無条件上書き）
- 失敗が**黙って握りつぶされる**（裸の `except: pass` が7箇所。うち1つは `DROP TABLE` を含む）
